# KMS

## KMS Key とは

* Key Container
  * 実体はHSM(Hardware Security Module)に格納されたHBK(HSM Backing Key) = Key material
  * 内部のHBKを複数保持している
* 暗号時には現在activeなHBKが利用される
* 暗号文にはどのHBKを利用したかのメタデータも埋め込んでいるため、復号時にどれを使えばいいかわかる


### KMS Key の種類

| 種類                 | 管理者 | 用途例                                                       |
|----------------------|--------|--------------------------------------------------------------|
| AWS managed key      | AWS    | S3, EBS 等の AWS サービスがデフォルトで使う (`alias/aws/s3`) |
| Customer managed key | ユーザ | アプリケーションが直接 API で使う。今回はこれ                |
| AWS owned key        | AWS    | ユーザからは見えない内部鍵                                   |


### Key Spec (鍵の暗号アルゴリズムによる分類)

KMS Key を作成するとき、鍵の暗号アルゴリズムを選ぶ。これが Key Spec。

| 分類             | Key Spec 例                                | 用途                                                |
|------------------|--------------------------------------------|-----------------------------------------------------|
| **対称暗号鍵**   | `SYMMETRIC_DEFAULT` (AES-256-GCM)          | データの暗号化/復号。1 つの鍵で暗号化も復号もできる |
| **非対称暗号鍵** | `RSA_2048`, `RSA_4096`, `ECC_NIST_P256` 等 | 公開鍵暗号。署名/検証、または暗号化/復号            |
| **HMAC 鍵**      | `HMAC_256`, `HMAC_384` 等                  | メッセージ認証コード生成/検証                       |

対称鍵 (Symmetric):
  同じ鍵で暗号化と復号を行う。
  鍵は KMS の外に出ない。Encrypt/Decrypt は全て KMS API 経由。
  (GenerateDataKey で鍵素材のコピーを取得する場合を除く)
  → 高速。大量データの暗号化向き。

非対称鍵 (Asymmetric):
  公開鍵 (public key) と秘密鍵 (private key) のペア。
  公開鍵は GetPublicKey API でダウンロード可能。
  秘密鍵は KMS の外に出ない。
  → 署名検証を KMS 外で行いたい場合、クロスプラットフォーム互換性が必要な場合に使う。

### Key Origin (鍵素材の出所)

KMS Key の鍵素材 (HBK) をどこで生成するかの選択。

| Origin                   | 説明                                     | ローテーション                           |
|--------------------------|------------------------------------------|------------------------------------------|
| **AWS_KMS** (デフォルト) | AWS KMS が HSM 内で鍵素材を生成          | 自動ローテーション可能                   |
| **EXTERNAL**             | ユーザが自分で生成した鍵素材をインポート | 自動ローテーション不可。オンデマンドのみ |
| **AWS_CLOUDHSM**         | CloudHSM クラスタで生成                  | 自動ローテーション不可                   |
| **EXTERNAL_KEY_STORE**   | 外部キーストア (XKS)                     | 自動ローテーション不可                   |

ローテーションの種類に直接影響する。

* `AWS_KMS` origin → 自動ローテーション対応 (KMS が内部で新しい HBK を生成できるから)
* `EXTERNAL` origin → 鍵素材をユーザが管理しているので、KMS が勝手に新しい鍵素材を生成できない
* `AWS_CLOUDHSM` / `EXTERNAL_KEY_STORE` → 同上

  
### KMSの識別子

| 識別子     | 形式            | 例                                                      | 特徴                                   |
|------------|-----------------|---------------------------------------------------------|----------------------------------------|
| Key ID     | UUID            | `123...90ab`                                            | 最もコンパクト。不変                   |
| Key ARN    | ARN             | `arn:aws:kms:ap-northeast-1:1...2:key/1234abcd-...`     | グローバル一意。クロスアカウントで必須 |
| Alias name | `alias/` prefix | `alias/app-key`                                         | 可変ポインタ。人間が読める             |
| Alias ARN  | ARN             | `arn:aws:kms:ap-northeast-1:123456789012:alias/app-key` | クロスアカウント + 可読性              |


### Alias

* Alias は KMS Key への可変ポインタ (mutable pointer)。
* いつでも別の Key に付け替え可能
* 1 つの Alias は 1 つの Key にしか紐付かない
* 1 つの Key には複数の Alias を付けられる (0 個でも可)
* `aws/` prefix は AWS 予約 (`alias/aws/s3` 等)
* Alias は独立したリソースで、独自の ARN を持つ

```
alias/idpstack-token-key ──→ KMS Key (UUID: 1234abcd-...)
                                │
                                ├── HBK v1 (2024-01 作成)
                                ├── HBK v2 (2025-01 ローテーション)
                                └── HBK v3 (2026-01 ローテーション) ← 現在アクティブ

```

## Key Rotation

### なぜ古い暗号文が復号できるのか

```
時系列:
  2024-01 │ KMS Key 作成。HBK v1 がアクティブ
          │ → データ A を暗号化 (HBK v1 で暗号化)
          │
  2025-01 │ 自動ローテーション発動。HBK v2 が生成されアクティブに
          │ → データ B を暗号化 (HBK v2 で暗号化)
          │ → データ A を復号 → OK! (HBK v1 はまだ保持されている)
          │
  2026-01 │ 自動ローテーション発動。HBK v3 が生成されアクティブに
          │ → データ A を復号 → OK! (HBK v1 はまだ保持されている)
          │ → データ B を復号 → OK! (HBK v2 もまだ保持されている)
```

1. KMS Key は全てのバージョンの HBK を永続的に保持する (削除しない限り)
2. 暗号文 (CiphertextBlob) にはどの HBK バージョンで暗号化したかのメタデータが含まれる
3. Decrypt 時、KMS はメタデータを読んで正しい HBK バージョンを自動選択する
4. アプリケーションからは完全に透過的 -- コード変更不要

# Rotation

| 種類                       | 対象                                   | 動作                                                       |
|----------------------------|----------------------------------------|------------------------------------------------------------|
| 自動ローテーション         | 対称暗号鍵 + AWS_KMS origin のみ       | スケジュール実行。デフォルト 365 日 (90-2560 日で設定可能) |
| オンデマンドローテーション | 対称暗号鍵 (AWS_KMS + EXTERNAL origin) | 手動トリガー。即時実行。自動ローテーションと同じ仕組み     |
| 手動ローテーション         | 非対称鍵、HMAC 鍵、CloudHSM/XKS 鍵     | 新しい KMS Key を作成し、Alias を付け替える                |

**自動/オンデマンド** は同じ KMS Key リソース内に新しい HBK バージョンを追加する。
Key ID も ARN も変わらない。アプリケーションは一切変更不要。

**手動ローテーション** は根本的に異なる。完全に別の KMS Key リソースを作る。

```
Before:  alias/my-key → Key A (Key ID: aaaa-...)
After:   alias/my-key → Key B (Key ID: bbbb-...)  ← Alias を付け替える
         Key A は Alias なしで残す (既存暗号文の復号用)
```

手動ローテーションが必要な鍵種別は、自動ローテーションの仕組み
(KMS が内部で新しい鍵素材を生成する) が使えないもの:

- 非対称鍵: 公開鍵が変わるとクライアント側の更新が必要で、透過的にできない
- HMAC 鍵: 同上 (検証側が新しい鍵を知る必要がある)
- EXTERNAL origin: 鍵素材をユーザが管理しており、KMS が勝手に生成できない
- CloudHSM / XKS: 鍵素材が外部システムにあり、KMS が制御できない


## 重要な注意点

ローテーションはデータ鍵を再暗号化しない
もし「データ鍵が漏洩した」場合は、ローテーションでは対処できない。
新しいデータ鍵を生成してデータを再暗号化する必要がある。
